4 数字化养殖大棚监控系统设计

本章在前文需求分析的基础上，对数字化养殖大棚监控系统的总体架构、网络结构、功能模块以及数据库结构进行设计。系统面向畜禽养殖大棚的环境监测、设备联动、异常告警和养殖档案管理等应用场景，采用“感知层、传输层、服务层、应用层”相结合的分层设计思想，实现环境数据采集、数据传输、业务处理、可视化展示和生产管理的一体化。

系统设计遵循可靠性、实时性、可扩展性和易维护性原则。在硬件侧，通过传感器与控制执行设备完成现场数据采集和设备动作；在软件侧，通过后端接口服务、数据库存储和前端监控页面实现数据处理与业务交互。各模块之间采用相对独立的设计方式，既保证系统功能完整，又便于后续根据养殖规模、设备类型和管理需求进行扩展。

4.1 系统总体架构设计

4.1.1 系统架构设计

数字化养殖大棚监控系统总体架构可划分为感知控制层、数据传输层、业务服务层和应用展示层四个部分。各层之间通过标准化数据格式和接口进行交互，从而降低系统耦合度，提高系统运行的稳定性和可维护性。

感知控制层主要由温湿度传感器、气体浓度传感器、单片机控制板、继电器模块以及风机、水帘等执行设备组成。该层负责采集养殖大棚内的温度、湿度、二氧化碳浓度、氨气浓度等环境参数，并根据上位机或后端下发的控制指令驱动相关设备执行动作。现场控制设备采用周期性采集方式获取环境数据，并将采集结果按照统一的数据格式上传。

数据传输层承担现场设备与后端系统之间的数据转发任务。系统通过串口桥接程序接收硬件端上报的数据，并调用后端 API 完成数据入库和业务处理。同时，桥接程序定期轮询后端待执行控制命令，将命令转换为硬件端可识别的 JSON 数据并下发至单片机。该设计使硬件设备与业务系统之间形成闭环通信，为自动控制和远程控制提供基础。

业务服务层是系统的核心处理部分，主要由 FastAPI 后端服务、业务服务模块、数据库访问层和缓存机制组成。后端服务负责接收环境数据、管理设备台账、执行告警判断、生成控制命令、维护养殖档案以及提供系统运行数据。系统采用 SQLAlchemy ORM 对数据库进行访问，减少直接编写 SQL 的复杂度，并提高数据模型与业务对象的一致性。

应用展示层主要由基于 Vue 的前端系统组成。该层面向养殖管理人员，提供环境监控、设备控制、告警处理、养殖档案、生产任务和系统管理等页面。用户可以通过浏览器查看实时环境指标、历史趋势、设备状态和告警信息，也可以进行手动控制、档案维护等操作。前端通过 HTTP 接口与后端服务通信，实现数据的实时刷新与业务操作。

系统总体架构可以描述如下：

感知控制层：传感器、STM32 控制板、继电器、风机、水帘等执行设备。

数据传输层：串口通信、桥接程序、JSON 数据协议。

业务服务层：FastAPI 接口、业务服务、SQLAlchemy、数据库和缓存。

应用展示层：Vue 前端、监控大屏、控制面板、档案与告警页面。

在该架构中，环境数据的流向为：设备采集、串口桥接、后端接口、数据库存储、前端展示；控制命令的流向为：前端操作或自动规则、后端生成命令、桥接程序轮询、硬件执行、状态回写。通过双向数据通道，系统能够实现监测、判断、控制和反馈的完整业务闭环。

4.1.2 网络结构设计

系统网络结构围绕现场设备、后端服务和前端客户端展开。现场设备主要部署在养殖大棚内部，负责接入传感器和执行设备；后端服务部署在本地服务器或管理主机上，负责统一接收数据和处理业务逻辑；前端客户端通过局域网访问后端接口，完成监控和管理操作。

在实际运行过程中，硬件控制板通过串口与桥接程序所在主机连接。桥接程序一方面读取串口数据，将采集到的环境参数封装后提交给后端；另一方面调用后端控制命令接口，获取待执行命令后通过串口发送至硬件端。后端服务对外提供 RESTful API，前端页面通过 HTTP 请求访问环境监测、设备控制、告警管理和档案管理等接口。

系统网络结构主要包括以下三类通信链路：

1. 现场采集链路。传感器与单片机之间通过硬件接口连接，单片机周期性读取采集值，并将数据组织成结构化消息。

2. 设备接入链路。单片机与桥接程序之间通过串口通信，桥接程序负责协议适配和数据转发。

3. 应用访问链路。前端浏览器通过局域网访问后端服务，后端服务通过数据库完成数据持久化，并将查询结果返回前端。

从网络功能角度看，系统采用本地化部署方式能够降低现场网络不稳定带来的影响，适合毕业设计验证和中小型养殖场景。若后续扩展为多棚区、多设备或云端集中管理模式，可在现有架构基础上引入 NB-IoT、4G/5G 或 MQTT 消息中间件，实现远距离设备接入和多节点集中管理。

4.2 功能模块设计

系统功能模块根据业务职责划分为环境监测模块、设备控制模块、告警管理模块和养殖档案模块。各模块之间既相互独立，又通过设备编号、环境数据和业务记录形成关联，共同支撑养殖大棚的数字化管理。

4.2.1 环境监测模块

环境监测模块是系统的基础模块，主要负责采集、接收、存储和展示养殖大棚内的环境参数。该模块监测的主要指标包括温度、湿度、二氧化碳浓度和氨气浓度。其中温湿度直接影响畜禽舒适度和生长状态，二氧化碳和氨气浓度则反映大棚通风状况和空气质量。

环境监测模块的主要功能包括：

1. 实时数据接收。后端提供遥测数据上报接口，接收设备编号、温度、湿度、气体浓度和采集时间等信息。

2. 数据合法性处理。系统对设备编号和环境指标进行基础校验，并对首次接入设备进行自动注册或关联。

3. 数据持久化存储。环境数据写入数据库中的环境数据表，为后续历史查询和趋势分析提供依据。

4. 最新数据展示。系统保存设备最新上报数据，前端页面可以展示当前环境状态和设备更新时间。

5. 历史趋势查询。用户可以按设备和时间范围查询历史数据，并通过图表观察环境变化趋势。

该模块的数据处理流程为：传感器采集环境参数后，控制板将数据发送至桥接程序；桥接程序调用后端遥测接口上传数据；后端完成数据解析、设备匹配、数据入库和告警判断；前端再通过接口获取最新数据和历史数据并进行可视化展示。

4.2.2 设备控制模块

设备控制模块用于实现对养殖大棚内执行设备的远程控制和自动联动控制。系统中的执行设备主要包括风机、水帘、补光设备等，控制方式包括人工手动控制和基于规则的自动控制。

手动控制主要面向管理人员的即时操作需求。用户在前端控制面板选择目标设备和控制指令后，前端将控制请求发送至后端。后端生成控制命令记录，并将命令状态设置为待执行。桥接程序轮询待执行命令，将命令下发至硬件控制板。硬件端执行后返回确认信息，桥接程序再将执行状态回写至后端。

自动控制主要面向环境异常或阈值触发场景。系统根据预设自动化规则对环境数据进行判断，例如当温度超过设定阈值时自动开启风机，当氨气浓度超过安全范围时联动通风设备。自动控制规则由触发指标、比较条件、阈值、目标组件、动作指令和优先级等字段组成，便于系统根据实际养殖环境进行调整。

设备控制模块的设计重点在于命令可追踪和状态可反馈。每条控制命令均记录设备编号、目标组件、命令类型、执行模式、执行状态、执行原因和操作人员等信息。通过命令日志，系统可以追溯设备动作来源，区分人工操作和自动规则触发，提高系统运行管理的透明度。

4.2.3 告警管理模块

告警管理模块用于对异常环境数据进行识别、记录、展示和处理。养殖大棚环境具有连续变化特征，当温度、湿度或有害气体浓度超过设定范围时，如果不能及时处理，可能影响养殖对象健康。因此，系统需要对关键环境指标进行实时判断，并向管理人员提供明确的告警信息。

告警管理模块的主要功能包括：

1. 告警规则判断。系统在接收环境数据后，对温度、湿度、二氧化碳浓度和氨气浓度等指标进行阈值判断。

2. 告警信息生成。当监测值超过阈值时，系统生成告警记录，保存告警类型、告警等级、阈值、实际值、告警描述和告警时间。

3. 告警状态管理。告警状态包括待处理、已确认和已解决等，管理人员可以在前端对告警进行确认和处理。

4. 告警列表展示。前端页面按照告警时间、等级和设备展示告警信息，便于用户快速定位异常来源。

5. 风险辅助决策。系统可结合待处理告警数量、告警等级和设备区域信息，形成风险摘要，为管理人员提供参考。

该模块与环境监测模块和设备控制模块紧密关联。环境数据是告警判断的输入，告警结果可以作为设备自动联动的依据。例如，当氨气浓度过高时，系统不仅生成告警记录，还可以联动开启风机，以降低棚内有害气体浓度。通过告警与控制联动，系统能够由被动监测转向主动调节。

4.2.4 养殖档案模块

养殖档案模块用于记录养殖批次、个体信息、免疫情况、健康状态和生产相关数据。传统养殖管理通常依赖纸质记录或人工表格，存在数据分散、查询不便和追溯困难等问题。通过建立电子化养殖档案，可以提高养殖过程的信息化水平，为后续生产分析和管理决策提供数据基础。

系统中的养殖档案分为批次档案和个体档案。批次档案用于记录同一批次养殖对象的编号、种类、数量、入栏时间、预计出栏时间、免疫记录、平均体重、饲料消耗和健康状态等信息。个体档案用于记录单个养殖对象的编号、品种、性别、出生日期、耳标、来源、体重、健康状态和免疫备注等信息。

养殖档案模块的主要功能包括：

1. 批次档案管理。支持新增、修改、删除和查询养殖批次信息，记录批次维度的生产数据。

2. 个体档案管理。支持维护单个养殖对象的基础信息和健康信息，实现更细粒度的追溯管理。

3. 档案统计展示。系统可统计当前有效批次数量、存栏数量、平均体重、饲料消耗和个体档案数量。

4. 变更记录维护。对于个体档案中的关键字段变化，系统可保存历史记录，便于追踪健康状态和生产状态变化。

养殖档案模块与环境监测模块之间具有业务关联。通过将不同棚区、设备和养殖批次进行对应，管理人员可以结合环境数据分析养殖状态变化，从而为通风、降温、补光、饲喂和免疫等管理措施提供依据。

4.3 数据库设计

4.3.1 数据库总体设计

数据库是系统实现数据持久化和业务追溯的核心。系统采用关系型数据库设计思想，通过设备表、环境数据表、告警信息表、控制命令表、自动化规则表和养殖档案相关表保存核心业务数据。后端使用 SQLAlchemy ORM 建立数据模型，将数据库表结构映射为 Python 对象，便于业务代码进行统一访问。

数据库设计遵循以下原则：

1. 实体清晰。按照用户、设备、环境数据、告警、控制命令和养殖档案等业务对象划分数据表。

2. 关系明确。通过外键建立设备与环境数据、设备与控制命令、用户与操作记录、批次档案与个体档案之间的关联。

3. 查询高效。对设备编号、采集时间、告警时间、档案编号等常用查询字段建立索引，提高实时查询和历史查询效率。

4. 支持扩展。表结构保留备注、状态、创建时间和更新时间等字段，便于后续增加业务字段和统计功能。

系统核心数据表关系如下：

用户表与设备表之间为一对多关系，一个用户可以管理多台设备；设备表与环境数据表之间为一对多关系，一台设备可以对应多条环境采集记录；设备表与告警信息表之间为一对多关系，一台设备可以产生多条告警记录；设备表与控制命令日志表之间为一对多关系，一台设备可以对应多条控制命令；养殖批次档案表与个体档案表之间为一对多关系，一个养殖批次可以包含多个个体档案；个体档案表与个体档案历史表之间为一对多关系，一个个体可以对应多条变更记录。

其中，设备表是环境监测、告警管理和控制命令之间的关联中心；养殖批次表和个体档案表构成生产档案数据的主体；用户表主要用于区分操作人员、管理人员和系统角色，为后续权限控制和操作审计提供基础。
