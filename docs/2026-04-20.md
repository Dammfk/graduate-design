## 1. 项目概述

本项目目标是开发一套联网的数字化养殖大棚监控系统，覆盖环境采集、设备控制、数据入库、前端展示、告警联动和生产管理等功能。

当前系统由四部分组成：

- 前端：Vue 3 + Vite + Pinia + Axios + ECharts
- 后端：FastAPI + SQLAlchemy + Pydantic + SQLite，Redis 可选
- 桥接：Python + pyserial + requests
- 硬件：STM32L433 + OLED + SHT3X + 继电器 + UART JSON 协议

当前运行环境：

- Windows 本机开发
- Python 3.12
- Node / npm
- STM32CubeIDE / STM32CubeProgrammer
- 当前串口：`COM5`
- 当前设备 ID：`DEVICE_001`
- 当前后端地址：`http://127.0.0.1:8000`

## 3. 仍未完善的重点问题

### 3.1 前端手动控制板子不稳定

这是当前最高优先级问题。

现象：

- 前端按钮能调用后端
- 后端能生成 `pending` 命令
- 桥接脚本能轮询并写串口
- 但板子不总是执行前端手动命令

当前判断：

- 上行 telemetry 链路稳定
- 下行 PC -> STM32 命令链路不稳定
- 更像硬件串口 RX 路径或板载串口路由问题
- 不能简单认定为前端按钮或后端 API 问题

下一步建议：

- 继续确认 KE1 板载 USB 串口是否真正支持 PC -> MCU 双向通信
- 如果板载链路不可靠，优先使用外接 USB-TTL 模块验证双向串口

### 3.2 自动规则仍是演示型规则

当前 `fan`、`cooling_pad`、`fill_light` 只是初步映射到继电器。

未完善点：

- 规则阈值未按真实养殖场景校准
- `fill_light` 是否应该参与当前硬件演示还未最终决定
- 自动规则和人工控制之间的优先级还未设计完整
- 板端手动覆盖与后端状态没有完全同步

### 3.3 前端展示仍是原型级

虽然前端能跑并显示数据，但仍有不少展示层问题：

- 继电器真实状态还没有作为 telemetry 字段上报
- 前端只能根据命令记录或推断状态展示控制结果
- CO2 和氨气目前 STM32 上报为 `null`
- 多模块页面已有框架，但并非全部完成真实业务闭环
- 错误提示、空状态、加载状态仍需统一打磨

### 3.4 数据模型和迁移方式还不正式

当前后端启动时直接 `create_all`，并在 `main.py` 里用 `ensure_runtime_schema()` 手工补字段。

这适合开发联调，但不是正式迁移方案。

后续建议：

- 引入 Alembic 或至少建立明确的数据库初始化脚本
- 清理历史测试数据
- 明确 SQLite 和 MySQL 的切换边界

### 3.5 Redis 是可选降级状态

Redis 连接失败时系统仍能运行。当前这是可接受的，但文档和部署说明应明确：

- Redis 不是当前必需依赖
- 缓存失败不应阻塞主流程
- 若部署生产环境，再决定是否强制要求 Redis

### 3.6 文档曾经写得过满

旧文档中有“100% 完成”“最小闭环完全打通”等表述。当前应理解为：

- telemetry 上报主链路已经打通
- 后端与前端基础展示可用
- 控制链路有过成功日志，但人工控制板子仍不稳定
- 项目仍处于联调完善阶段，不是最终完成状态

## 7. 接下来优先做什么

### 第一优先级：修通人工控制板子

目标：

```text
前端按钮 -> 后端 pending 命令 -> bridge 下发 -> STM32 执行 -> ACK -> 后端 success
```

建议先做：

- 用串口助手或最小脚本验证 `COM5 -> STM32` 是否稳定可收
- 若 `COM5` 下行不稳定，改用外接 USB-TTL 模块
- 在 STM32 收到命令时增加可见反馈，例如 OLED 短暂显示 `CMD OK`
- 在 bridge 中增加命令下发后的 ACK 超时标记

### 第二优先级：上报继电器真实状态

当前前端无法可靠知道继电器真实状态。  
建议 STM32 telemetry 增加：

```json
{
  "relay1_status": "ON",
  "relay2_status": "OFF"
}
```

然后后端模型、schema、前端控制面板同步支持。

### 第三优先级：整理控制规则

建议明确：

- 风机对应哪个继电器
- 水帘对应哪个继电器
- 补光灯是否参与当前硬件演示
- 自动规则和手动控制谁优先
- 手动控制后何时恢复自动模式

### 第四优先级：数据库迁移正规化

建议：

- 引入 Alembic
- 移除 `main.py` 中临时补字段逻辑
- 建立清晰的初始化和演示数据脚本

### 第五优先级：文档和演示流程

建议完善：

- 烧录步骤
- 启动步骤
- 验证 telemetry 步骤
- 验证前端页面步骤
- 验证人工控制步骤

- 数据采集和上报已经能跑
