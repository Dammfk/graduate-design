6 系统测试

本章对数字化养殖大棚监控系统进行测试说明，主要包括测试环境、功能测试和性能测试三部分。测试目标是验证系统在当前软硬件联调条件下，是否能够完成环境数据采集、串口桥接、后端入库、前端展示、控制命令生成和命令状态回写等核心功能。由于本项目处于毕业设计联调阶段，测试重点放在主业务链路可用性和系统连续运行稳定性上。

当前系统最稳定的链路为环境数据上报链路，即 STM32 采集数据后，通过 USART1 输出到 Windows 主机串口 COM5，serial_bridge.py 读取串口数据并转发给 FastAPI 后端，后端完成数据入库后由前端页面展示。控制下发链路已经具备前端、后端和桥接程序代码基础，但硬件端接收下行命令和 ACK 回传仍需要结合串口链路继续验证。因此，在测试结论中需要区分“软件链路已打通”和“硬件执行稳定性仍需完善”两类结果。

6.1 测试环境

测试环境由硬件环境和软件环境两部分组成。硬件环境主要用于采集现场环境数据和执行控制动作，软件环境主要用于运行后端服务、前端页面、数据库和串口桥接程序。系统测试采用本地联调方式进行，后端、前端和桥接脚本运行在同一台 Windows 主机上，硬件端通过 USB 串口与主机连接。

6.1.1 硬件环境

本系统硬件测试环境以 STM32 控制板为核心，控制板烧录 hardware/demo 工程固件。硬件端通过传感器采集温湿度数据，并通过串口输出 JSON 格式的遥测数据。当前联调中，温湿度采集、OLED 本地显示和 telemetry 上报链路已经能够稳定工作；CO2 和氨气传感器在当前阶段以接口和字段预留为主，部分测试数据可通过模拟值或 null 值进行联调。

硬件测试环境主要包括以下组成部分：

1. STM32 控制板：运行当前硬件演示固件，负责本地采集、OLED 显示、继电器控制和串口数据输出。

2. 温湿度传感器：用于采集大棚内温度和湿度，是当前硬件实测的主要环境数据来源。

3. OLED 显示模块：用于在硬件端显示温湿度等本地采集数据，便于测试人员判断硬件采集是否正常。

4. 继电器模块：用于验证风机等执行设备的开关控制逻辑。当前继电器基础控制代码已经具备，但前端人工控制到板子的稳定闭环仍需继续验证。

5. USB 串口连接：硬件端通过 USART1 连接到 Windows 主机，当前联调串口为 COM5，波特率为 115200。

6. 执行设备或模拟负载：用于观察继电器动作效果，可连接排风风机、水帘控制端或演示用指示负载。

硬件环境测试前需要确认板子上电正常，OLED 能显示采集数据，Windows 设备管理器中能够识别到串口号，且串口未被串口助手或其他程序占用。

6.1.2 软件环境

软件测试环境主要包括 Python 后端运行环境、Vue 前端运行环境、SQLite 数据库、可选 Redis 缓存和串口桥接程序。后端服务基于 FastAPI 实现，前端基于 Vue 3 实现，串口桥接程序使用 Python 编写。

软件环境主要配置如下：

1. 操作系统：Windows 开发环境。

2. 后端框架：FastAPI、SQLAlchemy、Pydantic、Uvicorn。

3. 数据库：当前联调使用 SQLite 数据库，数据库文件为 backend/greenhouse.db。

4. 缓存：Redis 为可选缓存，用于保存设备最新数据。若 Redis 连接失败，系统仍可降级为数据库查询，不影响 telemetry 入库主流程。

5. 前端框架：Vue 3、Vite、Axios、Pinia、ECharts。

6. 串口桥接：tools/serial_bridge.py，依赖 pyserial 和 requests。

7. 当前测试设备编号：DEVICE_001。

系统推荐使用 VS Code 任务 app: dev 启动，该任务会并行启动后端、前端和桥接脚本。也可以手动分别启动三个模块。后端启动命令为 python -m uvicorn main:app --host 127.0.0.1 --port 8000；前端启动命令为 npm run dev；桥接脚本启动命令为 python serial_bridge.py --port COM5 --baudrate 115200 --base-url http://127.0.0.1:8000 --device-id DEVICE_001。

测试前需要先访问 http://127.0.0.1:8000/health，确认后端返回 healthy；再访问前端页面 http://127.0.0.1:5173，确认页面能够正常打开；最后启动串口桥接程序，确认控制台输出“串口已打开: COM5 @ 115200”。

6.2 功能测试

功能测试主要验证系统核心业务是否能够按照设计要求运行。测试内容包括环境监测测试和控制功能测试。其中，环境监测测试重点验证数据采集、上报、入库和展示链路；控制功能测试重点验证前端命令、后端命令记录、桥接下发和硬件 ACK 回写流程。

6.2.1 环境监测测试

环境监测测试用于验证系统是否能够接收 STM32 上报的环境数据，并在后端和前端正确展示。测试过程中，首先启动后端服务、前端页面和串口桥接脚本，然后观察桥接程序是否能持续收到 STM32 输出的 telemetry JSON 数据。

测试输入示例为：

{"device_id":"DEVICE_001","temperature":28.4,"humidity":54.9,"co2_concentration":null,"ammonia_concentration":null}

桥接程序收到上述数据后，会将其转发到后端 POST /api/v1/telemetry/ 接口。若上报成功，控制台应出现“telemetry 已上传: DEVICE_001”日志。后端接收数据后，会根据 device_id 查询设备；若设备不存在，则自动注册设备；随后将温度、湿度、二氧化碳浓度、氨气浓度和时间写入 environment_data 表，并更新设备最近上报时间。

环境监测测试步骤如下：

1. 启动后端服务，访问 /health，确认服务状态为 healthy。

2. 启动前端页面，进入环境监测模块。

3. 启动串口桥接脚本，连接 COM5，波特率设置为 115200。

4. 观察桥接程序是否持续输出“串口接收”和“HTTP 上报”日志。

5. 调用 GET /api/v1/telemetry/latest/DEVICE_001，检查是否返回最新温湿度数据。

6. 调用 GET /api/v1/telemetry/history/DEVICE_001?hours=24，检查是否返回历史数据列表。

7. 在前端环境监测页面查看实时指标和历史趋势图是否随数据更新。

测试结果表明，telemetry 上报链路能够完成从 STM32 串口输出到 FastAPI 后端入库，再到前端页面展示的流程。温湿度数据能够在前端环境仪表中显示，历史数据能够通过 TrendChart 组件绘制趋势图。当前 CO2 和氨气字段在部分硬件测试中可能为 null，但后端和前端均已预留字段，能够支持后续真实传感器接入。

该测试的通过标准为：桥接脚本能够收到 JSON 数据；后端接口返回 success；数据库中新增 environment_data 记录；设备最新数据接口能够查询到数据；前端环境监测页面能够显示对应设备的最新温湿度和历史趋势。

6.2.2 控制功能测试

控制功能测试用于验证前端人工控制命令和后端自动控制命令是否能够生成，并通过桥接程序发送给 STM32。当前系统的控制对象主要包括 fan、cooling_pad 和 fill_light，其中 fan 对应排风风机，cooling_pad 对应水帘降温，fill_light 对应补光灯。

人工控制测试时，用户在前端设备控制页面选择 DEVICE_001 对应设备，并点击排风风机的 ON 或 OFF 操作。前端会调用 POST /api/v1/control/devices/{device_id}/commands 接口，后端根据目标组件和命令类型创建 control_command_logs 记录，命令初始状态为 pending。随后 serial_bridge.py 定时调用 GET /api/v1/control/devices/{device_id}/commands/pending 获取待执行命令。

桥接程序获取命令后，会将命令转换为发送给 STM32 的紧凑 JSON 格式，例如：

{"command_id":38,"target":"fan","command":"ON"}

如果串口写入成功，桥接程序会调用 PUT /api/v1/control/commands/{command_id}/status，将命令状态更新为 sent。硬件端执行命令后，理想情况下会回传 ACK，例如：

{"device_id":"DEVICE_001","type":"ack","command_id":38,"status":"ok"}

桥接程序收到 ACK 后，会将 ok 转换为 success，并再次调用命令状态接口完成状态回写。此时后端命令状态应变为 success，前端命令记录中也应能看到对应结果。

控制功能测试步骤如下：

1. 确认后端、前端和桥接程序均已启动。

2. 在前端控制页面选择目标设备 DEVICE_001。

3. 点击排风风机开启按钮，观察后端是否生成控制命令。

4. 观察桥接程序是否输出“串口发送”日志。

5. 若硬件端回传 ACK，观察桥接程序是否输出“命令回执已更新”日志。

6. 查看前端命令历史或数据库 control_command_logs，确认命令状态由 pending 更新为 sent 或 success。

测试结果表明，前端、后端和桥接程序之间的控制链路已经具备。前端点击控制按钮后，后端能够生成控制命令，桥接程序能够轮询到待执行命令并写入串口。在理想情况下，硬件端回传 ACK 后，命令状态能够更新为 success。

但当前项目也存在一个实际限制：前端人工控制到板子动作的稳定性仍受 PC 到 STM32 下行串口链路影响。如果测试中只出现“串口发送”日志而没有 ACK，说明前端和后端命令生成正常，桥接程序也已尝试下发，问题更可能出现在 COM5 到 STM32 的下行通信或硬件端命令解析环节。后续可通过外接 USB-TTL 模块、增加 OLED 命令提示和单独串口收发测试进一步定位。

6.3 性能测试

性能测试主要从响应时间和稳定性两个方面进行。由于本项目是毕业设计联调系统，测试重点不是高并发压力测试，而是验证在本地部署和单设备接入条件下，系统是否能够及时处理数据上报、查询请求和控制命令，并在连续运行过程中保持基本稳定。

6.3.1 响应时间测试

响应时间测试主要针对后端接口和前端页面交互进行。测试对象包括环境数据上报接口、最新数据查询接口、历史数据查询接口、控制命令创建接口和待执行命令轮询接口。

测试方法为：在后端服务启动后，使用浏览器、前端页面或接口测试工具多次访问相关接口，观察接口返回是否及时；同时在串口桥接程序运行过程中，观察 telemetry 上报日志和前端页面刷新情况。由于系统运行在本地环境，网络传输延迟较小，主要耗时来自后端数据处理、数据库读写和前端图表渲染。

主要测试接口包括：

1. POST /api/v1/telemetry/：验证环境数据上报、设备自动注册、数据入库、告警判断和控制命令生成的处理时间。

2. GET /api/v1/telemetry/latest/DEVICE_001：验证最新数据查询时间。该接口优先读取 Redis 缓存，缓存不存在时查询 SQLite 数据库。

3. GET /api/v1/telemetry/history/DEVICE_001?hours=24：验证历史数据查询和趋势图数据返回时间。

4. POST /api/v1/control/devices/DEVICE_001/commands：验证人工控制命令生成时间。

5. GET /api/v1/control/devices/DEVICE_001/commands/pending：验证桥接程序轮询命令的响应时间。

测试结果表明，在本地单设备联调条件下，环境数据上报接口能够及时返回处理结果，最新数据接口和历史数据接口能够满足前端页面刷新需求。前端切换设备或时间范围后，趋势图能够较快更新。控制命令创建后，桥接程序可在轮询周期内获取到 pending 命令。当前桥接程序默认轮询间隔为 2 秒，因此人工控制命令从后端生成到桥接程序发现命令之间，理论上会受到该轮询周期影响。

从系统实现角度看，影响响应时间的主要因素包括数据库数据量、历史查询时间范围、前端图表数据点数量、Redis 是否可用以及桥接程序轮询间隔。当前系统采用 Redis 缓存最新数据，可以减少最新数据查询对数据库的依赖；历史数据查询则依赖 environment_data 表中的 device_id 和 recorded_at 字段索引。对于当前毕业设计单设备和少量历史数据场景，系统响应时间能够满足监测展示和联调测试需要。

6.3.2 稳定性测试

稳定性测试用于验证系统在连续运行过程中是否能够持续接收环境数据、保存数据并保持前端显示正常。测试过程中，后端、前端和串口桥接程序同时运行，STM32 持续通过串口输出 telemetry 数据。测试人员观察桥接程序日志、后端接口状态、数据库记录增长和前端页面显示变化。

稳定性测试步骤如下：

1. 启动后端服务，确认 /health 正常。

2. 启动前端页面，进入环境监测模块。

3. 启动 serial_bridge.py，连接 COM5 和 DEVICE_001。

4. 保持系统连续运行一段时间，观察是否持续出现 telemetry 上报成功日志。

5. 定期访问最新数据接口和历史数据接口，确认数据仍可查询。

6. 观察前端页面是否持续显示最新数据和趋势变化。

7. 在运行期间尝试人工控制命令，观察命令生成、串口发送和 ACK 回写情况。

测试结果表明，当前系统在 telemetry 上报方面具备较好的连续运行能力。桥接程序能够持续读取串口数据并上传后端，后端能够持续写入 SQLite 数据库，前端页面能够显示最新环境数据。Redis 缓存异常不会中断主流程，因为后端在缓存读写失败时会捕获异常并继续使用数据库。

稳定性测试也发现当前系统存在需要继续完善的部分。第一，人工控制板子动作稳定性仍需重点验证，尤其是 COM5 到 STM32 的下行串口链路；第二，当前 telemetry 中继电器真实状态尚未完整上报，前端控制状态部分仍依赖命令日志或规则推断；第三，CO2 和氨气真实传感器接入仍需进一步完成，当前字段在部分联调中可能为 null；第四，当前数据库使用 SQLite，适合开发和毕业设计演示，若后续扩展为多设备长期运行，应考虑正式数据库迁移和数据清理策略。

总体来看，系统核心监测链路已经能够稳定支撑毕业设计演示需求，控制链路的软件部分已经打通，但硬件下行执行稳定性仍需继续优化。后续可从串口硬件连接、STM32 命令解析、ACK 机制、继电器状态上报和数据库迁移等方面继续完善系统稳定性。
